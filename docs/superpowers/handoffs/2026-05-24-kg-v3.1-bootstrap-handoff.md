## What v3.1 is

A bootstrap pipeline so end users can install the kg plugin **without a Go toolchain or CGO build**. Solves the friction surfaced when v3 was first dogfooded in Claude Code:

> /kg:kg-enrich pre-check failed — kg CLI not on PATH, no kg.db, manifest not in ~/.config/kg-extractor/

Two parts shipped together as v0.3.1:

1. **GitHub releases** via GoReleaser + GHA matrix (4 platforms: darwin/arm64, darwin/amd64, linux/amd64, linux/arm64). Each release publishes a `.tar.gz` containing `kg`, `kg-extractor`, `kg-extractor-tree-sitter`, plus `manifest.json` and `checksums.txt`.
2. **`kg-plugin/scripts/bootstrap.sh`** + smart SKILL pre-check that detects missing CLI, prompts user via `AskUserQuestion` to download v0.3.1 from GitHub, verifies SHA-256, extracts into `${KG_HOME:-$HOME/.config/kg}/`.

End-user UX after this lands:

```
/plugin marketplace add github:ggfarmco/kg
/plugin install kg@kg-graph
/kg:kg-enrich              # plugin offers to install kg CLI on first run; user says yes; done
```

## Scope decisions already made (don't relitigate)

These were brainstormed at the end of the v3 session — locked in for v3.1 MVP:

1. **GitHub repo:** `github.com/ggfarmco/kg` (new public). Created during Phase 2 task 4.
2. **Install location:** `${KG_HOME:-$HOME/.config/kg}` with env override. `bin/` for binaries, `extractor-plugins/tree-sitter/` for the tree-sitter manifest+symlink, `VERSION` file for upgrade detection.
3. **Bootstrap UX:** SKILL offers via `AskUserQuestion` ("Download v0.3.1 from github.com/ggfarmco/kg/releases? [y/N]"); on yes, runs `bash $CLAUDE_PLUGIN_ROOT/scripts/bootstrap.sh` inline.
4. **Versioning:** lock-step plugin↔CLI. `kg-plugin/.claude-plugin/plugin.json` `version: "0.3.1"` ≡ release tag `v0.3.1`. bootstrap.sh reads plugin's version, adds `v` prefix when constructing URL.
5. **Platforms:** 4 only — `darwin/arm64`, `darwin/amd64`, `linux/amd64`, `linux/arm64`. No Windows, no Alpine/musl, no FreeBSD.
6. **Distribution format:** one `.tar.gz` per platform with all 3 binaries + `manifest.json` + `README` + `LICENSE` inside. `checksums.txt` separate (auto-generated by GoReleaser).
7. **Verification:** SHA-256 only. No GPG signing in v0.3.1.
8. **Pre-check fallback order:** `command -v kg` → `${KG_HOME}/bin/kg` → `${CLAUDE_PLUGIN_ROOT}/../bin/kg` → `$(pwd)/bin/kg`. First executable wins. Same multi-location lookup applied across all SKILLs that call kg/kg-extractor.
9. **Bash deps for bootstrap.sh:** `bash`, `curl`, `tar`, `sha256sum` OR `shasum -a 256`, `uname`, `mktemp`, `jq`. All except `jq` are universally present; `jq` is documented as a prereq (Claude Code itself relies on jq in many skills so this is reasonable).

## Scope deferred (don't include in v3.1 MVP)

- Windows support — no demand, complicates GHA matrix
- musl/Alpine separate Linux builds — single glibc build per arch
- GPG release signing — HTTPS + SHA-256 sufficient
- Silent / hook-based auto-install — explicit consent via AskUserQuestion always
- Independent plugin/CLI versioning + compatibility matrix — lock-step removes the entire problem class
- `kg-bootstrap` Go binary alternative — pure bash works, no chicken-and-egg
- `kg uninstall` verb — `rm -rf ~/.config/kg/` is a one-liner
- Background upgrade nagging — next `/kg:kg-enrich` after a plugin update will offer it inline
- Marketplace-level version tracking — versions live in each plugin's own manifest
- jq fallback (grep/sed) — document jq as prereq; revisit if a user complains

## Caveats / things to watch out for

- **`bootstrap.sh` tar-extraction logic.** The spec's pseudo-code (`tar -xzf -C bin/`) is wrong — the tarball will also contain README and LICENSE which shouldn't end up in `bin/`. Real implementation: extract to tmp dir, then `mv` only the 3 binaries to `bin/` and `manifest.json` to `extractor-plugins/tree-sitter/`. Plan should explicitly call this out as the correct approach.
- **GoReleaser `archives.files` adds extra files but doesn't strip directory structure.** Make sure `dst: manifest.json` puts the manifest at archive root (next to binaries), not in a `plugins/tree-sitter/` subdirectory.
- **CGO on `ubuntu-24.04-arm`** is newer; may have toolchain quirks. Fallback: `ubuntu-22.04-arm` if the 24.04 image is missing essential build tools. Both should work; pin only after CI proves green.
- **`gh repo create` requires `gh` CLI auth.** If user hasn't run `gh auth login`, Phase 2 task 4 stalls. Document the requirement; the implementer can prompt user before invoking `gh repo create`.
- **First v0.3.1 release fixes are normal.** Don't treat first-release CI failures as showstoppers — GoReleaser configs typically need 2-3 iterations to land cleanly. Plan should include "iterate on workflow until green" as an explicit sub-step of task 5.
- **Plugin update sync delay.** After `/plugin update kg@kg-graph` pulls a new `plugin.json` with bumped version, the user's *next* `/kg:kg-enrich` triggers the upgrade prompt — there's no live notification. This is intentional (silent network = bad UX) but worth documenting in the SKILL's user-facing summary.
- **Lock-step versioning side effect.** README-only changes still require a CLI tag bump to keep plugin↔release in sync. Mitigation: bootstrap is idempotent on existing-version match, so plugin can bump to a new minor version that points at an existing release tag (manually verify the tag exists before pushing). Re-evaluate if this becomes confusing.
- **`jq` as bootstrap.sh prereq.** If a user lacks `jq`, bootstrap fails. Document in error message: `apt install jq` / `brew install jq`. Could add fallback `grep '"version"'` parse but adds maintenance burden.
- **Marketplace name `kg-graph` (not `kg`).** Locked in by issue #56043 workaround. Plugin name is still `kg`. Install command is `kg@kg-graph`. Don't rename to `kg` thinking it'll be cleaner — that re-triggers the bug.
- **No `gh` CLI fallback in bootstrap.sh.** Pure `curl` for download. Don't add `gh release download` path — too many auth edge cases. If GH ratelimits anonymous fetches in the future, document `GH_TOKEN` env var support as a future addition.
